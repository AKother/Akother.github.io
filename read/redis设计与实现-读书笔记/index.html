  • 以空字符串结尾（SDS函数自动完成添加和删除）–> 好处是可以重用C语言中的字符串函数库

SDS与C字符串的区别

  • 常量获取字符串长度，因为保存了len，C只能遍历
  • 杜绝缓冲区溢出（不管是直接分配还是拼接(sdscat))，因为保存了free，而C可能会假定内存够，实则溢出
  • 减少修改字符串时带来的内存重分配技术：C在截断或拼接时都要管理内存，否则可能会有内存溢出或者内存泄漏，而Redis实现如下：
    • 空间预分配：要对SDS空间扩展时，同时分配需要空间和额外空间
      • 修改后len<1MB: len = free
      • 修改后len>1MB: free = 1MB
    • 惰性空间：String中移除字符后不会立刻释放空间，由free记录（可能以后用到）（源码：用户再次调用该字符串，会调用释放空间的API）
  • **二进制安全：**C字符串有编码限制（不能保存图像、音频等二进制数据），但Redis用的buf[]则可以
  • Redis有需要时也可以重用C语言函数库（感觉这个不算区别）

解决键冲突

链地址法解决

rehash

当哈希表保存的键值对数量太多或者太少时，程序需要对哈希表的大小进行相应的扩展或者收缩。

步骤

  • 为ht[1]分配空间（根据使用空间分配）

    • 如果是扩展操作，ht[1]大小=第一个大于等于ht[0].used*2的2^n

    • 如果是收缩操作，ht[1]大小=第一个大于等于ht[0].used的2^n

      (这里描述的有点难受，实际上就是，如果扩3，要变为4；如果要扩9，要变为16，只能是2的整数倍)

  • 将ht[0]的所有键重新计算索引值，并将键值对一起放到ht[1]指定位置

  • 迁移完毕以后释放ht[0]，另ht[0]=ht[1]（新），再新建一个ht[1]

哈希表的扩展与收缩时机

扩展
  • 哈希表负载因子>=1，服务器没有执行BGSAVE或BGREWRITEAOF
  • 哈希表负载因子>=5，服务器正在执行BGSAVE或BGREWRITEAOF

也就是说只要扩展因子>=1就有可能实现哈希表扩展或收缩

收缩
  • 哈希表负载因子<=0.1，自动开始收缩

渐进性rehash

这也是实际上的用法，渐进式rehash中字典会同时用ht[0]和ht[1]（可能先去ht[0]找，之后再进ht[1]）

为了避免rehash对服务器性能造成影响，服务器不是一次性将ht[0]里面的所有键值对全部rehash到ht[1]，而是分多次、渐进式地将ht[0]里面的键值对慢慢地rehash到ht[1]。

  • 为ht[1]表分配空间
  • 在字典中维护索引计数器rehashidx，将值设为0时表示rehash开始
  • 每次增删改查，都会顺带将ht[0]在rehashidx索引上的所有键值对rehash到ht[1]，并将rehashidx+1
  • 某个时间点上ht[0]的所有值都会被rehash到ht[1]，此时令rehashidx=-1表示rehash操作完成

注意：

  • 新增key-value全都会被保存到ht[1]，这个步骤保证ht[0]只增不减，最终变为空表

第五章 跳跃表

跳跃表（skiplist）是一种有序数据结构，它通过在每个节点中维持多个指向其他节点的指针，从而达到快速访问节点的目的。平均O(logN)、最坏O(N)

很多程序用跳跃表来代替平衡树，因为性能差不多但跳表实现更简单

  • 当有序集合数量多，或有值是比较长的字符串时，跳表就被用作有序集合键的底层实现

第七章 压缩列表

压缩列表（ziplist）是列表键和哈希键的底层实现之一。

当一个哈希键只包含少量键值对，比且每个键值对的键和值要么就是小整数值，要么就是长度比较短的字符串，那么Redis就会使用压缩列表来做哈希键的底层实现。

第八章 对象

在前面的数个章节里，我们陆续介绍了Redis用到的所有主要数据结构，比如简单动态字符串（SDS）、跳表、双端链表、字典、压缩列表、整数集合等等。

Redis并没有直接使用这些数据结构来实现键值对数据库，而是基于这些数据结构创建了一个对象系统，这个系统包含字符串对象、列表对象、哈希对象、集合对象和有序集合对象这五种类型的对象，每种对象都用到了至少一种我们前面所介绍的数据结构。

六个数据结构 -> 五个对象

好处：

1.解耦，数据结构可以优化，上层对象可以扩展

2.上层数据结构在合适的情况下使用合适的实现，提高性能，优化内存

对象的类型与编码

  • Redis中保存的键值对，键只能是字符串对象，值可以是上述之一

数据库键空间

因为数据库的键空间是一个字典，所以所有针对数据库的操作，比如添加一个键值对到数据库，或者从数据库中删除一个键值对，又或者在数据库中获取某个键值对等，实际上都是通过对键空间字典进行操作来实现的

  • redisDb的dict字典保存了数据库中所有键值对

这些维护基本实现了Redis的大部分功能

当使用Redis命令对数据库进行读写时，服务器不仅会对键空间执行指定的读写操作，还会执行一些额外的维护操作，其中包括： ❑在读取一个键之后（读操作和写操作都要对键进行读取），服务器会根据键是否存在来更新服务器的键空间命中（hit）次数或键空间不命中（miss）次数，这两个值可以在INFO stats命令的keyspace_hits属性和keyspace_misses属性中查看。 ❑在读取一个键之后，服务器会更新键的LRU（最后一次使用）时间，这个值可以用于计算键的闲置时间，使用OBJECT idletime命令可以查看键key的闲置时间。 ❑如果服务器在读取一个键时发现该键已经过期，那么服务器会先删除这个过期键，然后才执行余下的其他操作，本章稍后对过期键的讨论会详细说明这一点。 ❑如果有客户端使用WATCH命令监视了某个键，那么服务器在对被监视的键进行修改之后，会将这个键标记为脏（dirty），从而让事务程序注意到这个键已经被修改过，第19章会详细说明这一点。 ❑服务器每次修改一个键之后，都会对脏（dirty）键计数器的值增1，这个计数器会触发服务器的持久化以及复制操作，第10章、第11章和第15章都会说到这一点。 ❑如果服务器开启了数据库通知功能，那么在对键进行修改之后，服务器将按配置发送相应的数据库通知，本章稍后讨论数据库通知功能的实现时会详细说明这一点。

过期键的删除策略

定时删除

用不到

  • 使用定时器，保证键一过期就立刻被删除

  • 对CPU不太友好

惰性删除

  • 取出键时对键做检查，如果过期了就删除
  • 对内存不友好，并且有些键可能永远都不会被删

定期删除

  • 每隔一段时间由程序执行一次删除过期键的操作
  • 难点在于操作执行的时长和频率

Redis的过期键删除策略

expireIfNeed

过期键的惰性删除策略由db.c/expireIfNeeded函数实现，所有读写数据库的Redis命令在执行之前都会调用expireIfNeeded函数对输入键进行检查

Redis：惰性删除 + 定期删除

activeExpireCycle

每当Redis的服务器周期性操作redis.c/serverCron函数执行时，activeExpireCycle函数就会被调用

  • 分多次遍历服务器中的各个数据库，从数据库的expires字典中随机检查一部分键的过期时间，并删除其中的过期键。
  • current_db会记录上一次检查进度
  • 全部检查完了就将current_db重置为0

AOF、RDB和复制功能对过期键的处理

生成RDB文件

  • save或besave创建RDB时，会先进行检查，RDB中不会保存过期键

载入RDB文件

  • 如果是主服务器模式：RDB中过期键不会被载入
  • 如果是从服务器模式：RDB中所有键都会被载入(过期键需要等待主服务器的同步命令来删除)

从AOF文件写入

  • AOF不会因为过期键产生影响
  • 过期键被删后，程序会向AOF中追加del命令表示该键已被删除

AOF重新写

  • 已过期键会被忽略，不会被重写到AOF

复制

  • 当服务器运行在复制模式下时，从服务器的过期键删除动作由主服务器控制：
    • 主服务器在删除一个过期键之后，会显式地向所有从服务器发送一个DEL命令，告知从服务器删除这个过期键。
    • 从服务器在执行客户端发送的读命令时，即使碰到过期键也不会将过期键删除，而是继续像处理未过期的键一样来处理过期键。
    • 从服务器只有在接到主服务器发来的DEL命令之后，才会删除过期键。

可能会有实时一致性问题，过期键仍然存在于从服务器；但可以保证最终一致性

写入与同步

Redis的服务器进程就是一个事件循环（loop），这个循环中的文件事件负责接收客户端的命令请求，以及向客户端发送命令回复，而时间事件则负责执行像serverCron函数这样需要定时运行的函数。

  • 结束事件循环前，调用flushAppendOnlyFile
  • 此时考虑是否将aof_buf中内容写入和保存到AOF文件
  • 这个考虑取决于appendfsync配置选项(always,everysecond,no)
  • 么appendfsync选项的默认值为everysec（每秒同步）

数据载入

因为AOF文件里面包含了重建数据库状态所需的所有写命令，所以服务器只要读入并重新执行一遍AOF文件里面保存的写命令，就可以还原服务器关闭之前的数据库状态。

  • 会有一个fake client，伪客户端没有网络连接

重写

随着时间推移，AOF体积可能过大，并且AOF只需要记录最新数据即可，因此Redis提供了AOF重写功能。

Redis服务器可以创建一个新的AOF文件来替代现有的AOF文件，新旧两个AOF文件所保存的数据库状态相同，但新AOF文件不会包含任何浪费空间的冗余命令，所以新AOF文件的体积通常会比旧AOF文件的体积要小得多。

Redis服务器是一个事件驱动程序，服务器需要处理以下两类事件：

  • 文件事件（file event）：Redis服务器通过套接字与客户端（或者其他Redis服务器）进行连接，而**文件事件就是服务器对套接字操作的抽象。**服务器与客户端（或者其他服务器）的通信会产生相应的文件事件，而服务器则通过监听并处理这些事件来完成一系列网络通信操作。
  • 时间事件（time event）：**Redis服务器中的一些操作（比如serverCron函数）需要在给定的时间点执行，**而时间事件就是服务器对这类定时操作的抽象。

文件事件

Redis基于Reactor模式开发了自己的网络事件处理器：这个处理器被称为文件事件处理器（file event handler）

Reactor模式（反应器设计模式）

  • Redis采用单线程Reactor模式
  • 由一个非阻塞线程处理并发请求
  • 并将请求提交到一个或多个服务处理程序的事件设计模式

文件事件处理器

file event handler

虽然文件事件处理器以单线程方式运行，但通过使用I/O多路复用程序来监听多个套接字，文件事件处理器既实现了高性能的网络通信模型，又可以很好地与Redis服务器中其他同样以单线程方式运行的模块进行对接，这保持了Redis内部单线程设计的简单性。

  • 如果一个套接字可读可写，服务器将优先处理读套接字（连接或命令请求）
  • 常用的处理器：连接应答、命令请求、命令回复
  • I/O多路复用程序同时监听多个套接字
  • 并根据套接字目前执行的任务关联不同事件处理器
  • 套接字准备执行操作（accept, read, write, close）等操作时，相关的文件事件会产生
  • I/O多路复用程序会将套接字放进队列中，有序单个的传递给事件处理器，并且只有上一个事件处理完后，才会继续分派套接字
  • 这些不同的事件处理器则会处理这些事件

I/O多路复用底层实现

Redis的I/O多路复用程序的所有功能都是通过包装常见的select、epoll、evport和kqueue这些I/O多路复用函数库来实现的

完整的客户端与服务器连接例子

建立连接过程

  • Redis客户端向服务器发起连接，产生AE_READABLE事件
  • 连接应答处理器为客户端创建套接字、状态、以及一个AR_READABLE事件
  • 并将AR_READABLE事件与命令请求处理器关联

处理命令过程

  • Redis客户端向服务器发送命令请求，产生AE_READABLE事件
  • 命令请求处理器接受事件，读取命令内容并将其交给相关程序执行
  • Redis客户端尝试读取命令执行结果，产生AE_WRITEABLE事件
  • 服务器将AE_WRITEABLE事件与命令回复处理器关联
  • 命令回复处理器接受事件，并将命令回复内容全部写入到套接字
  • 服务器解除AE_WRITEABLE事件与命令回复处理器关联

连接不关闭、请求处理器不关闭、但是回复处理器会在发完回复以后就取消关联

时间事件类型

  • 定时事件（目前Redis没用到）
  • 周期性事件

时间事件组成部分

  • id（唯一ID）
  • when（毫秒+事件到达时间）

实现原理

  • 每次有时间事件处理器运行时就遍历整个链表

第十五章 复制

在Redis中，用户可以通过执行SLAVEOF命令或者设置slaveof选项，让一个服务器去复制（replicate）另一个服务器，我们称呼被复制的服务器为主服务器（master），而对主服务器进行复制的服务器则被称为从服务器（slave）

旧版复制方式

  • 执行slaveof [master_ip] [master_port]
  • 设置slaveof选项
1
2

127.0.0.1: 12345> SLAVE 127.0.0.1 6379
# 然后12345会成为6379的从服务器

旧版复制功能实现

  • 同步 sync

  • 命令传播 command propagation：主服务器将可能使得主从不一致的命令发送给从服务器，从服务器执行

旧版复制功能缺陷

  • 断线重连后复制，从服务器都会接收很多不必要的数据（主服务器没必要执行besave然后创建从开始到现在的RDB文件，这不是非做不可）
  • 主服务器的bgsave虽然非阻塞（子线程），但是也会降低主服务器性能
  • 从服务器载入大量数据时是完全阻塞的
  • 发送RDB文件也会占用大量网络资源

新版复制方式

  • 执行PSYNC命令
    • 完整重同步
    • 部分重同步

  • 只需要发送断线期间的数据
  • 传输内容从RDB变成了命令

部分重功能实现

组成部分

  • 主服务器的复制偏移量和从服务器复制偏移量

  • 主服务器的复制积压缓冲区（FIFO）

  • 服务器运行ID（相同->部分重同步；不同->完整重同步）

PSYNC命令实现

两种调用方式

  • 从服务器之前从来没复制过：执行PSYNC?-1，调用完整重同步
  • 从服务器之前已经复制过：调用PSYNC [runid] [offset]（主服务器判断执行完整重同步还是部分重同步）

复制的实现过程

如果一切顺利⬇️

  • 从执行slaveof [master_id] [master_port]
  • 主建立套接字连接，主为从专门创建处理复制工作的文件事件处理器（RDB、命令）
  • 从发送PING命令，主回发PONG命令
  • 身份验证（可选，根据从的masterauth，校验主从密码是否相同）
  • 从发送端口信息，主记录端口信息
  • 从请求同步
  • 命令传播

心跳检测

方式

  • 从以每秒一次频率向主服务器检测
  • 方式是发送命令REPLCONF ACK [replication_offset]

作用

  • 检测主从网络连接状态
  • 辅助实现min-slaves选项：防止主在不安全情况下执行写命令

第十六章 Sentinel

Sentinel（哨岗、哨兵）是Redis的高可用性（high availability）解决方案：由一个或多个Sentinel实例（instance）组成的Sentinel系统（system）可以监视任意多个主服务器，以及这些主服务器属下的所有从服务器，并在被监视的主服务器进入下线状态时，自动将下线主服务器属下的某个从服务器升级为新的主服务器，然后由新的主服务器代替已下线的主服务器继续处理命令请求。

  • Sentinel字典保存除了自己以外，监视相同主服务器的Sentinel

  • 由订阅频道，监视同一个主服务器的Sentinel可以发现监视相同主服务器的Sentinel

只是命令连接，不会有订阅连接，订阅连接只是用于发现新的Sentinel

检查主观下线状态

  • 每秒一次，Sentinel向已有命令连接的所有实例（主服务器、从服务器、Sentinel）发送ping
    • 有效恢复：PONG, LOADING, MASTERDOWN
    • 无效回复：其他回复或没有回复
  • 如果没有回复，判定该实例主观下线

检查客观下线状态

  • 确定主服务器主观下线后，轮询监视该主服务器的Sentinel
  • 确定有足够数量的Sentinel也认为该主服务器下线
  • 判定该服务器为客观下线（只是这个Sentinel判定）
  • 对服务器进行故障转移

数量条件：quorum参数值

不同Sentinel判断客观下线的条件可能不同

选举领头Sentinel

当一个主服务器被判断为客观下线时，监视这个下线主服务器的各个Sentinel会进行协商，选举出一个领头Sentinel，并由领头Sentinel对下线主服务器执行故障转移操作。

“无论你设置要多少个Sentinel同意才能判断一个服务器失效，一个Sentinel都需要获得系统中多数Sentinel的支持， 才能发起一次自动故障迁移”

Sentinel系统选举领头Sentinel的方法是对Raft算法的领头选举方法的实现

  • 发生在有主服务器被客观下线时
  • 选出来的领头Sentinel要做故障转移操作

规则

  • 所有在线的Sentinel都可以是leader
  • 选举完以后不论结果，所有Sentinel的configuration epoch(可以理解为计数器)会+1
  • configuration epoch中，所有Sentinel都有一次设置某个Sentinel为leader的机会，设置后不可更改，每个epoch只会有一个leader
  • 每个发现主服务器客观下线的Sentinel都会要求其他Sentinel将自己设置为leader
  • 设置leader是先到先设置，Sentienl A向Sentinel B发送设置请求，那么Sentinel B之后会拒绝其他Sentinel的请求
  • 某个Sentinel被半数以上Sentinel设置为leader，则该Sentinel为leader
  • 没选出来就一段时间后重选

故障转移

大概内容：

  • 在已客观下线主服务器属下的所有从服务器中，挑出一个从服务器，将它变成主服务器
  • 让其余所有从服务器都改为复制新的主服务器
  • 将已下线服务器设置为从服务器，当它重新上线时，就是新主服务器的从服务器

挑选新的主服务器过程

先删除

做减法的过程，leader Sentinel会将从服务器保存到列表中，一步步过滤

  • 删除处于下线或者断线状态的从服务器
  • 删除最近五秒内未回复过leader Sentinel的INFO命令的从服务器
  • 删除所有与已下线主服务器断开超过down-after-milliseconds*10毫秒的从服务器

再根据优先级挑选

  • 选复制偏移量大的从服务器（数据会更新）
  • 如果一样大，选运行ID最小的

第十七章 集群

Redis集群是Redis提供的分布式数据库方案，集群通过分片（sharding）来进行数据共享，并提供复制和故障转移功能。 本节将对集群的节点、槽指派、命令执行、重新分片、转向、故障转移、消息等各个方面进行介绍。

节点

  • 一个Redis集群通常由多个节点组成

  • 节点起初都是互相独立的，各自在只包含自己的集群中

  • 连接节点的命令：CLUSTER MEET [ip] [port] （握手）

之后，节点A会将节点B的信息通过Gossip协议传播给集群中的其他节点，让其他节点也与节点B进行握手，最终，经过一段时间之后，节点B会被集群中的所有节点认识。

启动节点

  • 节点和在单机Redis中做的内容没啥区别

槽指派

Redis集群通过分片的方式来保存数据库中的键值对：集群的整个数据库被分为16384个槽（slot），数据库中的每个键都属于这16384个槽的其中一个，集群中的每个节点可以处理0个或最多16384个槽。

当数据库中的16384个槽都有节点在处理时，集群处于上线状态（ok）；相反地，如果数据库中有任何一个槽没有得到处理，那么集群处于下线状态（fail）。

  • 集群通过分片方式保存键值对
  • 分片是指集群的整个数据库被分为16384个槽
  • 每个键都属于16384个槽中的一个
  • 每个节点最多可以处理0～16384个槽
  • 16384个槽全部在被处理，集群才是上线状态
  • 计算key属于哪个槽：CRC16(key) & 16384

指派命令

  • CLUSTER ADDSLOTS [slot] [slot …]

ASK错误

  • 情况：源节点向目标节点迁移时，一部分键值对在源节点，一部分在目标节点

  • 解决方法：

ASKING命令

ASKING命令唯一要做的就是打开发送该命令的客户端的REDIS_ASKING标识

在一般情况下，如果客户端向节点发送一个关于槽i的命令，而槽i又没有指派给这个节点的话，那么节点将向客户端返回一个MOVED错误

但是，如果节点的clusterState.importing_slots_from[i]显示节点正在导入槽i，并且发送命令的客户端带有REDIS_ASKING标识，那么节点将破例执行这个关于槽i的命令一次。

  • REDIS_ASKING标识是一次性的

ASK错误和MOVED错误区别

  • MOVED错误代表槽的负责权已经从一个节点转移到了另一个节点：在客户端收到关于槽i的MOVED错误之后，客户端每次遇到关于槽i的命令请求时，都可以直接将命令请求发送至MOVED错误所指向的节点，因为该节点就是目前负责槽i的节点。
  • 与此相反，ASK错误只是两个节点在迁移槽的过程中使用的一种临时措施：在客户端收到关于槽i的ASK错误之后，客户端只会在接下来的一次命令请求中将关于槽i的命令请求发送至ASK错误所指示的节点，但这种转向不会对客户端今后发送关于槽i的命令请求产生任何影响，客户端仍然会将关于槽i的命令请求发送至目前负责处理槽i的节点，除非ASK错误再次出现。
  • MOVED：永久措施，直接转发。请求的键不在当前节点，重定向到该槽指派的节点
  • ASK：临时措施，没有影响。请求的键在当前节点，但是可能被分派到新节点，重定向到新节点，并向节点发送一个ASKING命令

复制与故障转移

  • 节点分为主节点和从节点
  • 主节点用于处理槽
  • 从节点用于复制某个主节点 / 被复制主节点下线时，代替下线主节点处理命令
  • 替换主节点和哨兵模式很像，主从交换
  • 一个节点称为从节点并开始复制主节点这个消息会广播
  • 定时发送PING检测其他节点是否在线

可以类比一下Sentinel判断主服务器下线的过程：

  • Sentinel判断一个主服务器下线，会有一个主观下线到客观下线的过程，Sentinel A认为主服务器下线，先标记为主观下线，然后向其他Sentinel 发送消息，如果包括自己在内超半数的Sentinel认为该主服务器下线了，那么Sentinel A 就标记主服务器为客观下线，并向其他Sentinel发现消息，选自己为最后执行故障转移的操作的领头Sentinel。

首先知道集群模式下的所有节点都保存了其他节点的状态信息，并且会通过gossip协议消息来同步。

  • 集群模式下的redis主从，其他主节点都同时充当数据库和Sentinel的角色，主节点A认为主节点B下线了，A标记B为疑似下线状态，这里可以类比主观下线；然后A将这个消息广播给其他主节点，其他主节点在自己的节点list里找到被疑似下线的B节点，添加A标记B疑似下线的报告。当超半数的主节点都认为节点B疑似下线，那么最先检测到自己节点列表中的B的疑似下线报告超半数的节点会标记B为下线状态，可以类比客观下线；然后该节点会将这个状态消息同步给其他节点。
  • 故障转移的过程：需要在已下线的主节点B的从节点中选出一个来继承B的财产，做主节点。
  • B的从节点得知B下线，都会抢着替代B做主节点，所以会向其他主节点发送消息，选自己的下一个主节点。这个投票过程和Sentinel 选出新主服务器的过程类似，都是对raft的实现，最终都是想选出一个状态良好、数据完整的从服务器，作为新主服务器。

故障转移步骤

  • 选取在复制的从节点之一
  • 被选中的从节点执行SLAVEOF no one，成为新的主节点
  • 新的主节点撤销已下线主节点的所有槽指派，并指派给自己
  • 新的主节点广播一条PONG消息，告知自己已是主节点
  • 新的主节点开始执行命令请求，故障转移完成

选取新的主节点

前面的sentinel是内部投票，即候选人和投票人都是sentinel，而这里相当于外部投票，候选人是失效主节点的所有从节点，投票人是除失效主节点外的所有主节点

sentinel leader和主节点的选举都是基于Raft算法

  • 集群的配置纪元是一个自增计数器，它的初始值为0。
  • 当集群里的某个节点开始一次故障转移操作时，集群配置纪元的值会被增一。
  • 对于每个配置纪元，集群里每个负责处理槽的主节点都有一次投票的机会，而第一个向主节点要求投票的从节点将获得主节点的投票。
  • 当从节点发现自己正在复制的主节点进入已下线状态时，从节点会向集群广播一条CLUSTERMSG_TYPE_FAILOVER_AUTH_REQUEST消息，要求所有收到这条消息、并且具有投票权的主节点向这个从节点投票。
  • 如果一个主节点具有投票权（它正在负责处理槽），并且这个主节点尚未投票给其他从节点，那么主节点将向要求投票的从节点返回一条CLUSTERMSG_TYPE_FAILOVER_AUTH_ACK消息，表示这个主节点支持从节点成为新的主节点。
  • 每个参与选举的从节点都会接收CLUSTERMSG_TYPE_FAILOVER_AUTH_ACK消息，并根据自己收到了多少条这种消息来统计自己获得了多少主节点的支持。
  • 如果集群里有N个具有投票权的主节点，那么当一个从节点收集到大于等于N/2+1张支持票时，这个从节点就会当选为新的主节点。
  • 因为在每一个配置纪元里面，每个具有投票权的主节点只能投一次票，所以如果有N个主节点进行投票，那么具有大于等于N/2+1张支持票的从节点只会有一个，这确保了新的主节点只会有一个。
  • 如果在一个配置纪元里面没有从节点能收集到足够多的支持票，那么集群进入一个新的配置纪元，并再次进行选举，直到选出新的主节点为止。

消息

Sentinel是通过命令连接和订阅连接

集群中的节点通过发送和接收消息来进行通信，常见的消息包括MEET、PING、PONG、PUBLISH、FAIL五种。

  • MEET：请求节点加入到当前集群
  • PING：检测在线
  • PONG：回复在线 / 告知其他节点自己已经变成主节点（故障转移成功）
  • FAIL：判断某节点已下线（广播所有节点，不用Gossip是因为Gossip有延迟）
  • PUBLISH：收到PUBLISH后会执行这个命令，并再广播一条PUBLISH消息，所有收到消息的节点会执行相同的PUBLISH命令

Gossip协议

  • Redis集群中的各个节点通过Gossip协议来交换各自关于不同节点的状态信息，其中Gossip协议由MEET、PING、PONG三种消息实现

  • meet用于将新节点邀请进集群，ping用于探测集群其他节点是否正常活跃，pong用于确认回复meet/ping消息

  • Gossip协议是集群扩散，多次传播达成集群一致性

第十八章 发布与订阅

Redis的发布与订阅功能由PUBLISH、SUBSCRIBE、PSUBSCRIBE等命令组成。

通过执行SUBSCRIBE命令，客户端可以订阅一个或多个频道，从而成为这些频道的订阅者（subscriber）：每当有其他客户端向被订阅的频道发送消息（message）时，频道的所有订阅者都会收到这条消息。

除了订阅频道之外，客户端还可以通过执行PSUBSCRIBE命令订阅一个或多个模式，从而成为这些模式的订阅者：每当有其他客户端向某个频道发送消息时，消息不仅会被发送给这个频道的所有订阅者，它还会被发送给所有与这个频道相匹配的模式的订阅者。

频道的订阅和退订

  • 所有频道的订阅关系保存在pubsub_channels字典，key是频道，value是记录订阅该频道的客户端的链表

  • 订阅，SUBSCRIBE [channel] ：添加链表 / 添加链表节点

  • 退订，UNSUBSCRIBE [channel] ：删除链表节点 / 如果链表为空则也删除键

模式的订阅与退订

  • 所有模式的订阅关系保存在pubsub_patterns，就是一个列表，每个节点都包含pubsubPattern结构

  • pubsubPattern记录被订阅的模式和订阅的客户端

  • 订阅模式 PSUBSCRIBE

  • 退订模式 PUNSUBSCRIBE

发送消息

  • 命令 PUBLISH [channel] [message]
  • 将message发送给channel频道所有订阅者
  • 将message发送给 与channel频道匹配的模式 的订阅者

查看订阅信息

  • 查看被订阅的频道 PUBSUB CHANNELS [pattern]（可选）
  • 查看订阅者数量 PUBSUB NUMSUB[channel-1 channel-2…channel-n]
  • 查看服务器被订阅模式数量 PUBSUB NUMPAT

先草率的记一下重点回顾

  • Redis服务器在启动时，会对内嵌的Lua环境执行一系列修改操作，从而确保内嵌的Lua环境可以满足Redis在功能性、安全性等方面的需要。
  • Redis服务器专门使用一个伪客户端来执行Lua脚本中包含的Redis命令。
  • Redis使用脚本字典来保存所有被EVAL命令执行过，或者被SCRIPT LOAD命令载入过的Lua脚本，这些脚本可以用于实现SCRIPT EXISTS命令，以及实现脚本复制功能。

  • SCRIPT FLUSH命令会清空服务器lua_scripts字典中保存的脚本，并重置Lua环境。
  • SCRIPT EXISTS命令接受一个或多个SHA1校验和为参数，并通过检查lua_scripts字典来确认校验和对应的脚本是否存在。
  • SCRIPT LOAD命令接受一个Lua脚本为参数，为该脚本在Lua环境中创建函数，并将脚本保存到lua_scripts字典中。
  • 服务器在执行脚本之前，会为Lua环境设置一个超时处理钩子，当脚本出现超时运行情况时，客户端可以通过向服务器发送SCRIPT KILL命令来让钩子停止正在执行的脚本，或者发送SHUTDOWN nosave命令来让钩子关闭整个服务器。
  • 主服务器复制EVAL、SCRIPT FLUSH、SCRIPT LOAD三个命令的方法和复制普通Redis命令一样，只要将相同的命令传播给从服务器就可以了。
  • 主服务器在复制EVALSHA命令时，必须确保所有从服务器都已经载入了EVALSHA命令指定的SHA1校验和所对应的Lua脚本，如果不能确保这一点的话，主服务器会将EVALSHA命令转换成等效的EVAL命令，并通过传播EVAL命令来获得相同的脚本执行效果。

    i = (i & 0x0F0F0F0F) + ((i>>4) & 0x0F0F0F0F));
    // 4
    i = (i*(0x01010101) >> 24);
}
  • 步骤1计算出的值i的二进制表示可以按每两个二进制位为一组进行分组，各组的十进制表示就是该组的汉明重量。
  • 步骤2计算出的值i的二进制表示可以按每四个二进制位为一组进行分组，各组的十进制表示就是该组的汉明重量。
  • 步骤3计算出的值i的二进制表示可以按每八个二进制位为一组进行分组，各组的十进制表示就是该组的汉明重量。
  • 步骤4的i*0x01010101语句计算出bitarray的汉明重量并记录在二进制位的最高八位，而＞＞24语句则通过右移运算，将bitarray的汉明重量移动到最低八位，得出的结果就是bitarray的汉明重量。

第二十三章 慢查询日志

Redis的慢查询日志功能用于记录执行时间超过给定时长的命令请求，用户可以通过这个功能产生的日志来监视和优化查询速度。

  • slowlog-log-slower-than：超过多少微秒会被记录

  • slowlog-max-len：最多保存几条

  • 慢查询日志的存储方式是链表

  • 新的日志会被添加到表头

第二十四章 监视器

  • 通过执行MONITOR命令，客户端可以将自己变为一个监视器，实时地接收并打印出服务器当前处理的命令请求的相关信息

  • 每当一个客户端向服务器发送一条命令请求时，服务器除了会处理这条命令请求之外，还会将关于这条命令请求的信息发送给所有监视器

  • 监视器的存储方式也是链表，存在redisServer的monitors中
