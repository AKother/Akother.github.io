从Paxos到ZooKeeper - 读书笔记

第一章 分布式架构

分布式环境的问题

  • 通信异常
  • 网络分区（脑裂）
    • 只有部分节点之间可以正常通信，出现小集群
    • 小集群会各自有首，独立完成本来需要一整个集群才能完成的事情
  • 节点故障

分布式特点

  • 分布性
  • 对等性
  • 并发性
  • 缺乏全局时钟

从ACID到CAP/BASE

分布式系统中，原本的ACID很难保证，因此转向了CAP/BASE理论

CAP理论

  • C：Consistency
  • A：Avaliability
  • P：Partition tolerance

无法同时满足3个要求，P是一定要保证的，因此往往会在C和A之间寻求平衡

一致性
  • 在多个节点之间读取更新，希望对每个节点做的操作结果都是相同的
  • 强一致性：对某个数据更新以后，所有用户都可以读取到最新的值，则为强一致性
可用性
  • 用户的请求会在有限的时间内返回结果
  • 有限的指标是系统设定的运行结果，各个系统都有不同
分区容错性
  • 发生网络分区时，仍然对外保证一致性和可用性

BASE理论

基本可用、软状态、最终一致性

  • Basically Avaliable
    • 允许响应时间变长
    • 允许功能上的损失
  • Soft state
    • 系统数据的中间状态，不影响可用性
  • Eventually consistent
    • 经过同步后，所有数据都达到一致状态
      • 因果一致性
      • 读己之所写
      • 会话一致性
      • 单调读一致性
      • 单调写一致性

第二章 一致性协议

用于处理分布式一致性问题，可以理解为分布式系统中的事务保证

著名的解决方式有

  • 2PC
  • 3PC
  • Paxos

2PC

Two-Phrase Commit

目前广泛使用的流程

执行流程

  • 阶段一 提交事务请求（类似投票阶段）：
    • 1.事务询问
      • 协调者向所有参与者发送事务内容，询问是否可以执行事务提交操作，并等待所有参与者的响应
    • 2.执行事务
      • 每个节点都执行这个事务，会将Undo和Redo信息记录到事务日志中
    • 3.参与者反馈响应
      • 参与者向协调者返回事务操作的结果
  • 阶段二 执行事务提交：
    • 如果收到的全部是Yes
      • 1.协调者向参与者发出Commit请求
      • 2.参与者执行事务Commit操作
      • 3.参与者向协调者反馈提交结果（Ack）
      • 4.收到全部反馈后，完成事务
    • 如果有收到一个No / 没有收到全部的反馈
      • 1.协调者向参与者发出Rollback请求
      • 2.参与者执行Rollback（根据Undo）
      • 3.参与者向协调者反馈回滚结果（Ack）
      • 4.收到全部反馈后，中断事务

优点

  • 简单方便

缺点

  • 同步阻塞：其他参与者需要等待未反馈的参与者
  • 单点问题；协调者挂了则挂了
  • 容错机制不完善：任何一个节点失败都会导致整个事务的失败

3PC

Three-Phase Commit

执行流程

  • 阶段一：CanCommit
    • 1.事务询问
      • 协调者发送canCommit请求，询问是否可以执行事务
    • 2.参与者反馈结果
      • 参与者如果认为可以，则反馈Yes（注意这里只是认为，而2PC是执行了事务内容的）
  • 阶段二：PreCommit
    • 如果所有参与者都是Yes
      • 1.发送预提交请求
        • 协调者发送preCommit请求，并进入Prepared阶段
      • 2.事务预提交
        • 参与者执行事务内容，并记录Undo和Redo信息到事务日志中
      • 3.参与者反馈响应
        • 如果执行没问题则返回Yes
    • 如果有No / 未收到全部参与者反馈
      • 1.发送中断请求
        • 协调者向所有参与者发送abort请求
      • 2.中断事务
        • 参与者接收到abort或者等待协调者请求超时，都会自行中断事务执行
  • 阶段三：DoCommit
    • 如果所有参与者都是Yes
      • 1.协调者向参与者发出doCommit请求
      • 2.参与者执行事务doCommit操作
      • 3.参与者向协调者反馈提交结果（Ack）
      • 4.收到全部反馈后，完成事务
    • 如果有收到一个No / 没有收到全部的反馈
      • 1.协调者向参与者发出abort请求
      • 2.参与者执行abort（根据Undo）
      • 3.参与者向协调者反馈回滚结果（Ack）
      • 4.收到全部反馈后，中断事务
    • 如果在该阶段中参与者没有收到协调者的doCommit / abort，等待超时后都会自行提交事务

优点

  • 降低参与者阻塞范围，避免了阶段三的单点故障

缺点

  • 阶段三的超时处理方式会导致数据不一致

Paxos算法

  • Proposer
  • Acceptor
  • Learner

摘录自JavaGuide

  • 提议者（Proposer）：也可以叫做协调者（coordinator），提议者负责接受客户端的请求并发起提案。提案信息通常包括提案编号 (Proposal ID) 和提议的值 (Value)。
  • 接受者（Acceptor）：也可以叫做投票员（voter），负责对提议者的提案进行投票，同时需要记住自己的投票历史；
  • 学习者（Learner）：如果有超过半数接受者就某个提议达成了共识，那么学习者就需要接受这个提议，并就该提议作出运算，然后将运算结果返回给客户端。

  • Paxos引入“过半”概念：少数服从多数
  • Paxos支持分布式节点角色转换，避免单点故障问题

第四章 ZooKeeper与Paxos

ZooKeeper应用场景

  • 数据发布/订阅
  • 负载均衡
  • 命名服务
  • 集群管理
  • Master选举
  • 分布式锁
  • 分布式协调/通知

特性

  • 顺序一致性

  • 原子性

  • 单一视图（数据模型是一致的 –> 但是数据不一定一致）

  • 可靠性

  • 最终一致性

  • 高性能

  • 可以构建集群

基本概念

集群角色

  • Leader
  • Follower
  • Observer

会话

  • 客户端与服务的的一次TCP长连接
  • 提供心跳检测机制

ZNode

  • ZooKeeper的数据单元
  • 四类节点
    • 持久节点
    • 持久顺序节点
    • 临时节点
    • 临时顺序节点
  • 由父节点维护顺序节点的自增数，该数会在顺序节点被创建时自动添加在节点名后

版本

  • ZooKeeper会为每一个ZNode维护Stat数据结构
  • 内容包括
    • version (ZNode版本)
    • cversion (当前ZNode子节点版本)
    • aversion (当前ZNode的ACL版本)

Watcher

  • 允许在ZNode上注册Watcher
  • 指定事件触发时，会通知客户端

ACL

Access Control Lists策略，ZNode采用该方式控制权限

  • CREATE
  • READ
  • WRITE
  • DELETE
  • ADMIN：设置子节点ACL的权限

为什么选择ZooKeeper

  • 免费、源码开放
  • 性能、易用性、稳定性都很优良

ZAB协议

ZooKeeper Atomic Broadcast

ZooKeeper原子消息广播协议

核心

  • 所有事务都由Leader（全剧唯一）处理
  • Leader会将事务转换为Proposal，发给集群中的所有Follwer
  • Leader等待所有Follwer的反馈
  • 如果超过半数的Follwer都支持，Leader会让所有Follwer都执行Commit，即将前一个Proposal进行提交

崩溃恢复和消息广播是ZAB协议的两种基本模式

崩溃恢复

  • Leader崩溃则会重新选举Leader
  • 选举得到Leader并且有过半的Follwer与其完成状态同步后，退出恢复模式，进入消息广播模式
  • 当新服务器加入集群后，如果发现有Leader在消息广播，则会自发进入数据恢复模式，与其进行数据同步

消息广播

  • 类似二阶段提交
  • 基于具有FIFO特性的TCP协议
  • 略有不同
    • 消息广播中没有中断逻辑，并不需要等待全部Follwer的响应，过半就好了

第六章 ZooKeeper的典型应用场景

数据发布订阅 / 配置中心

原理

发布者将数据发布到ZooKeeper的节点上，供有需要的订阅者进行订阅，实现动态抓取

ZooKeeper采用推拉结合的模式：客户端向服务端注册自己需要关注的节点，如果该节点有变化，则发送Watcher事件通知给客户端（推），客户端再拉取最新的数据（拉）

实现方式

  • 选取一个节点用于配置存储
  • 每台机器初始化阶段都去该节点读取配置中心，并注册一个该配置节点的Watcher，及时接收更改
  • 如果收到Watcher事件通知，客户端需要拉取最新配置

命名服务

  • 基于顺序节点生成分布式唯一ID

分布式协调/通知

  • 基于Watcher的注册和异步通知实现
  • 可以用于异步数据复制、数据变化通知、监控
  • 心跳检测
  • 机器的工作进度汇报
  • 系统调度

集群管理

  • 集群中机器的上线和下线操作和管理
  • 知道当前集群中的机器数目
  • 对集群中每台机器运行状态进行数据收集

为什么选ZooKeeper做集群管理

  • Watcher监听节点，如果变化可以立刻收到通知
  • 机器都是临时节点，如果机器自动断线，则其与服务器之间的会话会失效，临时节点也会被自动删除，因此不需要做多余的下线操作

Master选举

比如单独让某个机器执行耗时很长的计算过程，需要选一个Master

  • Master选举（分布式问题）：面对特殊的任务，需要从集群中选出一台进行处理

用ZooKeeper（基于强一致性，节点一定全局唯一）实现的思路：

  • 选取根节点
  • 很多台机器同时向这个节点创建子节点
  • 创建成功的那台唯一机器就是Master
  • 可以为该Master注册一个Watcher，一旦Master挂了则重新进行Watcher选举

分布式锁

排他锁

写锁，只能有一个事务对其进行操作

  • 用一个数据节点来表示一个锁

  • 获取锁：create()，只能保证一个客户端创建锁 –> 获取到了锁

    • 如果宕机，则临时节点被移除，不会导致死锁

  • 释放锁：delete()

共享锁

读锁，只能进行读取操作。其他事务也只能对该对象加共享锁，直到该对象上的所有共享锁都被释放

判断读写顺序

（这里的读写是比较公平的，先来先获锁）

  • 注册监听（只监听前者，避免羊群效应）
    • 读：对比自己序号小的最后一个写请求注册Watcher
    • 写：对比自己序号小的最后一个节点注册Watcher
  • 对于读请求
    • 如果没有比自己序号小的节点，或是比自己序号小的节点都是读请求，那么拿到共享锁
    • 如果比自己序号小的节点中有写请求，则进入等待
  • 对于写请求
    • 如果自己不是序号最小的子节点，则进入等待
