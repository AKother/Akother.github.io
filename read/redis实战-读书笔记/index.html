Redis实战-读书笔记

该书深入浅出地介绍了 Redis 的字符串、列表、散列、集合、有序集合等五种结构， 并通过文章聚合网站、cookie、购物车、网页缓存、日志、计数器、IP 所属地址查询程序、自动补全、分布式锁、计数信号量、任务队列、消息队列、搜索程序、广告定向程序、社交网站等一系列实用示例展示了 Redis 的用法。

第一章 初识Redis

文章排名实战

目的

单纯用Redis实现文章排行榜，根据文字的发布时间和投票数量计算出评分，根据评分对文章进行排名和展示。

数据结构

内容 数据结构类型 图例 备注
文章信息 Hash 根据文章ID获取文章内容
根据发布时间排序的文章 ZSet 快速获取时间排序
根据评分排序的文章 ZSet 快速获取评分排序
已投票用户 Set 每个文章都有这个集合，判断已投票过的用户，文章发布一周后自动删除

对文章进行分组

意义

  • ZSet可以用于[快速获取排名]功能
  • ZSet的ZINTERSTORE命令可以取集合和有序集合的交集，如果将集合中放入特定topic的文章，再与score结合，可以实现[获取特定主题中的文章，按评分排序]

登录cookie

Cookie类型 简介 缺点
签名Cookie 用户信息（可能包含签名）都存在Cookie中 可能忘记，安全漏洞
令牌Cookie 存储一段随机数字作为令牌，服务器根据该令牌获得用户信息 服务器中要存更多信息

目的

大多数关系型数据库都比较慢

利用Redis优化Cookie功能

实现功能

校验cookie

更新cookie
  • 每次用户浏览页面都会更新cookie
  • 更新内容：时间戳，浏览过的商品

定期清理cookie
  • 移除cookie的同时要移除对应的用户信息

集群模式

计数信号量

计数信号量用于让多个客户端同时访问相同信息（读锁），还可以限制一个资源最多被多少个进程访问

与锁的区别：进程获取锁失败可能会等待，而获取信号量失败则会立刻返回（fail-fast）

超时构建信号量

同上面分布式锁，使用EXPIRE即可

有序集合构建信号量

  • 成员是进程的唯一标识符
  • score是尝试获取信号量时间戳
获取信号量

  • 排名低于可获取的信号量总数则取得信号量
  • 获取失败则移除ZSet中自己的标识符
释放信号量

存在问题
  • 分布式情况会受主机时间影响，各个进程系统时间未必相同 –> 不公平的锁

公平信号量

为了避免系统时间带来的不公平影响，对上诉信号量模式修改如下

  • 新增规则：增加一个计数器，最先对计数器自增的客户端可以获得信号量
  • 新增“信号量拥有者”ZSet，成员为进程唯一标识符，计数器的值为score，通过检查rank来判断进程是否取得信号量

消除竞争条件

  • 自增计数器和将标识符添加到ZSet这两个操作并非原子的，存在竞争条件

解决方法：

  • 加一个短暂超时锁（保证原子性），成功获取锁后，再执行信号量获取操作

任务队列

  • 普通队列：list
  • 优先级队列：可以用ZSet；也可以用书中的“多个队列”，然后用BLPOP LIST1 LIST2 .. LISTN TIMEOUT ，只有优先级高的队列为空，才能弹出优先级低的元素
  • 延迟队列：ZSet，其中score为时间戳
