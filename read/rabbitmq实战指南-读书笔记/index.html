RabbitMQ实战指南 - 读书笔记

内容简介

  • 基础篇首先介绍 RabbitMQ 的基本安装及使用方式，方便零基础的读者以最舒适的方式融入到RabbitMQ 之中。其次介绍 RabbitMQ 的基 本概念，包括生产者、消费者、交换器、队列、绑定等。之后通过 Java 语言讲述了客户端如何与 RabbitMQ 建立(关闭〉连接、声明(删除)交换器、队列、绑定关系，以及如何发送和消费消息等。
  • 进阶篇讲述 RabbitMQ 的 TIL、死信、延迟队列、优先级队列、 RPC、消息持久化、生产端和消费端的消息确认机制等内容，以期 读者能够掌握 RabbitMQ 的使用精髓。本书中间篇幅主要从RabbitMQ 的管理、配置、运维这三个角度来为读 者提供帮助文档及解决问题的思路。
  • 高阶篇主要阐述 RabbitMQ 的存储机制、流控及镜像队列的原理，深入 地讲述RabbitMQ 的一些实现细节，便于读者加深对 RabbitMQ 的理解。本书还涉及网络分区的概念，此内 容可称为魔鬼篇，需要掌握前面的所有内容才可理解其中的门道。本书最后讲述的是 RabbitMQ 的一些扩展内 容及附录，供读者参考之用。

实战部分没有实践过，只了解了概念，本书等工作后会再次补齐笔记

第一章 RabbitMQ简介

作用

  • 存储消息
  • 解耦
  • 削峰
  • 可恢复性
  • 顺序保证
  • 缓冲
  • 异步通信

RabbitMQ的特点

  • 可靠性：持久化、传输确认、发布确认
  • 灵活的路由：内置交换器实现路由，也可以插件实现自己的路由
  • 扩展性：多个RabbitMQ可以组成一个集群
  • 高可用：队列可以在集群中的机器上设置镜像，以作备用
  • 多种协议：AMQP、STOMP、MQTT
  • 多语言客户端：Java、Python、Ruby、PHP、C#
  • 管理界面：用户可以在管理页面上监督和管理消息、集群节点
  • 插件机制：发布扩展和用户自己编写

第二章 RabbitMQ入门

RabbitMQ模型介绍

  • 隐去中间细节，整体上来说是生产者和消费者模型
  • 消息 = [消息体，标签(记录交换机、路由键)]，标签在进入队列时被丢弃，生产者只会收到消息体

运转过程

连接模型

  • 多个生产者会复用一个Connection，再建立属于自己的channel
  • Connection是TCP连接，开销大，RabbitMQ采用No-blocking IO的方法，可以减少性能开销

交换机

  • 生产者将消息投入到交换机中，交换机再分配给队列
  • RoutingKey：路由键，用于指定消息的路由规则
  • 四种类型
    • fanout：广播该交换机收到的消息到所有队列
    • direct：只发送给完全匹配的队列
    • topic：发送给符合匹配规则的队列（正则）
    • headers：基本不用，是根据消息内容中的headers属性匹配

队列

  • 多个消费者订阅同一个队列时，消息会被轮询平分

AMQP协议

介绍

  • 是应用层协议的开方标准
  • 基于此协议的客户端和消息中间件可以传递消息，不受产品、开发语言等条件的限制

结构

  • Module Layer：协议最高层，提供客户端调用命令
  • Session Layer：中间层，客户端命令->服务器，服务器结果->客户端，会提供可靠传输机制和错误处理
  • Transport Layer：传输层，类似网络七层架构里面的，提供帧的处理、信道复用、错误检测、数据表示

第四章 RabbitMQ进阶

提升数据可靠性有以下一些途径: 设置 mandatory 参数或者备份交换器 (immediate 参数己被陶汰);设置 publisher config机制或者事务机制;设置交换器、队列和消息都为持久 化;设置消费端对应的 autoAck 参数为 false 井在消费完消息之后再进行消息确认。

本章不仅 介绍了数据可靠性的一些细节，还展示了 RabbitMQ 的几种己具备或者衍生的高级特性，包括 TTL、死信队列、延迟队列、优先级队列、 RPC 功能等，这些功能在实际使用中可以让相应应 用的实现变得事半功倍 。

消息不可达时

  • mandatory参数
    • true：找不到队列时，消息返回给生产者
    • false：消息被直接丢弃
  • immediate参数
    • true：找到队列，但没有消费者时，消息返回给生产者

备份交换器

  • 如果不想消息直接被丢弃（比如找不到队列），可以将未被正确路由的消息存在备份交换器中
  • 需要与备份交换机绑定队列的routingkey匹配，否则也会丢失。为了方便可以设置routing type = fanout（也就是不需要匹配，广播即可）

过期时间

  • 消息和队列都可以设置TTL
  • 设置消息的TTL
    • 设置队列：队列中的消息有相同的TTL
      • 消息过期方式：一过期则清除，因为TTL都一样
    • 设置消息：某条消息有自己TTL
      • 消息过期方式：惰性删除，等要发消息了再看看该消息过期没有，如果过期了才删，因为TTL不一样
  • 设置队列的TTL
    • TTL到达时队列自动被删除
  • TTL超过时，消息会变为死信

死信队列

Dead-Letter-Exchange：死信交换机

  • 消息变成死信的情况
    • 消息被拒绝
    • 消息过期
    • 队列满了
  • 可以监听死信队列并且做一些业务处理

延迟队列

  • 消息进入队列后，需要等待一段时间才能发送给消费者
  • 应用场景
    • 下单，30分钟内完成支付
    • 想让智能设备在指定时间内工作，让消息在延迟队列中定时发送

优先级队列

  • 优先级高的消息优先处理

RPC实现

Remote Procedure Call

  • 回调队列：接收response消息（客户端会监听）
  • correlationId：关联request和RPC调用返回的response

持久化

  • 交换机、队列、消息持久化

生产者确认

  • 提供生产者确认自己的消息有没有正常发送到服务器的数据
    • 事务机制
      • 事务提交成功则说明消息正确送达
      • 但是挺浪费性能的，会阻塞
    • 发送方确认机制
      • 消息（每个消息都包含唯一ID）成功投递后，RabbitMQ会发送唯一ID目前给生产者
      • 可以单条消息确认，可以批量确认（可能导致批量重传）
      • 和TCP很像
  • 这两个机制不能共存
  • 只能保证消息被正确发送到MQ的交换器

消费端要点

  • 消息分发方式：轮询
    • 也可以设置每个消费者最多在确认前收到多少消息
    • 类似滑动窗口
  • 消息顺序性：收到消息和发布消息的顺序是一致的
    • 打乱消息顺序性的情况
      • 多个生产者情况下，有消息重发
      • 消息优先级
      • 消息TTL不一致 / 延迟队列

消息传输保障

  • 最多一次
  • 最少一次
  • 恰好一次

第七章 RabbitMQ运维

集群节点类型

  • 内存节点
  • 磁盘节点
    • 单节点集群只有磁盘节点（否则断电丢失）
  • 集群中至少需要一个磁盘节点
    • 如果所有磁盘节点都崩溃，集群可运行，但是无法做任何更改

查看服务日志

  • 日志级别：none, error, warning, info, debug
  • 查看日志方式
    • 人工查看
    • 交换器（eg: amq.rabbitmq.log ( topic )）

单节点故障恢复

  • 机器掉电故障，节点处于stop状态
    • 其他节点上执行rabbitmqctl forget_cluster_node {nodename}移除故障节点
    • 再删除故障节点的Mnesia
    • 最后重启，将故障节点作为新节点重新加入集群
    • （如果盲目重启，可能会导致网络分区问题）
  • 网卡故障
    • 关闭该机器节点上的故障RabbitMQ进程
    • 可能有网络分区现象，需要手动恢复
  • 服务进程异常
    • 考虑风险可控，可控的话一般直接重启即可

集群迁移

  • 元数据重建
  • 数据迁移
  • 客户端连接切换

自动化迁移

  • 三个部分：客户端、集群、ZooKeeper

第八章 跨越集群的界限

Federation

  • 使RabbitMQ在不同的Broker节点之间传递消息，不需要建立集群

  • 用到了联邦交换器（Federated Exchange）和AMQP

  • 应用场景：如果发送方和接收方距离比较远，发送方在收到确认回馈前需要忍受很长一段时间的网络延迟

  • 其实意思就是中间建一个转发站（内部Broker存放数据 + AMQP拉取数据）。发送方如果与接收方距离很远，那么发送方借助这个转发站，可以快速的收到反馈消息，减少延迟。信息到内部Broker以后就交给它慢慢转发

  • 中间完全可以建立多个联邦交换器

  • 联邦队列

    • 联邦队列之间可以建立Federation link，下游的联邦队列消息如果消费完毕，可以拉取上游的关联队列
    • 提供负载均衡，联邦队列中的消息会转向有多余消费能力的一方

Shovel

  • 可靠持续的将Broker A中队列的消息转发到Broker B的交换器（Broker A 和 Broker B 可以相同）
  • 优势
    • 松耦合
    • 广域网
    • 高度定制
  • 中间会根据需要建立队列 / 交换机
  • Shovel相比Federation，处在更低一层（涉及到broker中的队列和交换机）

Federation / Shovel 与集群的区别

案例：消息堆积的治理

  • 方案A：丢弃

  • 方案B：增加消息消费能力

  • 方案C：可以设置Shovel，将队列中的堆积消息交给另一个集群，当队列中消息数有足够容量时，再移回来

第九章 RabbitMQ高阶

存储机制

队列的结构

队列消息状态

如果消息投递的队列是空的，并且此时有消费者，则消息不会进入队列，而是直接投递给消费者

消息索引是全局KV，RabbitMQ所有消息都存在这里

进入队列以后，消息的状态可能不断变化

  • 消息状态
    • alpha: 消息内容和消息索引都在内存
    • beta: 消息内容在磁盘，消息索引在内存
    • gamma: 消息内容在磁盘，消息索引在磁盘和内存
    • delta: 消息内容和消息索引都在磁盘

惰性队列

  • 尽可能将消息存储到磁盘中
  • 如果消费者需要，才将其加载到内存
  • 这样可以减少消耗，但会增加IO

流控

Erlang进程通过消息传递来通信，每个进程都有自己的进程邮箱，如果大量消息发送往进程，可能会导致内存溢出，此时就需要流控

  • 基于信用证算法的流控机制
  • 监视各个进程的邮箱
  • 当进程负荷量过大的时候拒绝接收上游消息，而上游进程邮箱此时也会开始堆积信息，当堆积到一定量时也会拒绝接受，最后会使网络数据包接收进程阻塞，从而达到暂缓接收的目的

镜像队列

  • 将队列镜像到集群中的其他Broker

  • 如果集群中的一个节点失效，则会自动切换到镜像节点，以保证肯定性

  • 不管消费者绑定master还是slave，消息总是发往master，同时也会发往所有slave，master会执行命令，同时将命令执行结果广播给所有slave，master或slave再将结果返回给消费者

  • 此处的master和slave都是针对queue，而队列可以均匀散布在集群的各个broker以实现负载均衡（真正的负载均衡是相对机器而言）

  • 如果主节点失效，slave全部断开，选举资历最老的slave为新master（如果未同步，会有消息丢失问题）

  • 一个broker中的主从结构

  • 多个broker构成集群结构

  • 镜像队列最后一个停止的节点是master

  • 必须是master先启动

  • 如果slave先启动，会等master 30s， 如果30s内没启动则slave自行停止

镜像队列的结构

  • 通过组播方式同步
    • GM：所有节点要么都收到，要么都收不到
    • GM原理：master和slave会形成一个组，不同的镜像队列形成不同的组，操作命令从GM发出后最终会回到GM，此时GM便知道完成了全部同

第十章 网络分区

网络分区情况

  • 出现网络分区时，不同分区的节点都认为别的分区节点挂了
  • 如果是集群发生网络分区，那么会各自为政
  • 如果设置了镜像队列，牵扯到了多个节点，每个分区又都出现了一个master，会出现一些异常情况

RabbitMQ为什么引入网络分区

  • 如果出现网络波动或故障，导致某个节点异常，这个异常会导致整个调用链阻塞
  • 需要用网络分区来把异常节点剥离出来
  • 是否阻塞会通过一个超时判定机制来判断
